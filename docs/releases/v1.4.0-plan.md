# v1.4.0 Release Plan

## Summary
v1.4.0 is the release where GFC starts becoming sustainable without turning into a bait-and-switch. The goal is to keep the forecast creation experience that users already trust fully available for free while introducing optional accounts, premium hosted cloud storage, and privacy-first product metrics for the official hosted service.

This release replaces the old roadmap assumption that v1.4.0 would be WarnGen-focused. WarnGen is still part of GFC's future, but the immediate priority is building a durable foundation for hosted accounts, billing, sync, and release operations before the scope expands further.

**Target ship window:** April 4-5, 2026 weekend

## Non-Negotiables
- GFC remains open source.
- Core forecasting features remain free.
- Accounts are optional, not mandatory.
- Local-first workflows remain intact for non-account users.
- Admin metrics stay anonymous and aggregate-only.
- Premium pays for the hosted service layer, not for core drawing, discussion, export, or verification tools.

## Locked Product Decisions
- **Authentication provider:** Firebase Auth
- **Supported auth methods:** Google and email/password
- **Cloud provider:** Firebase Storage and Firebase-managed data services
- **Billing provider:** Stripe
- **Free synced data:** account profile and app settings
- **Premium synced data:** hosted forecast cycles and discussions
- **Monthly premium price:** $3/month
- **Annual launch promo:** $25/year for the first two weeks after launch
- **Standard annual price after promo:** $30/year
- **Beta rollout:** closed beta first for highly active users and trusted friends before public launch

## Free vs Premium Boundary

### Free
- Forecast drawing and editing
- Local auto-save
- Local cycle history
- JSON import/export
- ZIP package export
- Forecast discussions
- Verification workflows
- Profile and settings sync across devices for signed-in users

### Premium
- Hosted cloud save/load for forecast cycles
- Hosted cloud save/load for discussions as part of the saved cycle payload
- Multi-device cloud library for premium-backed cycles
- Background sync for cloud-backed work on the official hosted deployment

### Explicitly Not in Scope for Premium
- Paywalling local forecast creation
- Paywalling verification
- Paywalling export/import
- Locking signed-out users out of the app

## Architecture Decisions

### Hosted Feature Flags
The hosted experience must be controlled by environment-driven capability checks:

- **No Firebase config:** app runs in local-only mode
- **Firebase enabled, Stripe disabled:** accounts and free settings sync are available, premium UI stays off
- **Firebase and Stripe enabled:** full hosted product is available

### Self-Host Fallback
The open-source repo must support a self-host capable path:

- Operators can bring their own Firebase and Stripe configuration
- Operators can also run the app with hosted features disabled and keep a local-only workflow
- Official hosted-service assumptions must be isolated behind config, not hardcoded into the app flow

### Server Responsibilities
The `server/` package evolves from analytics-only support into the hosted-service companion API for:

- Stripe checkout session creation
- Stripe billing portal session creation
- Stripe webhook handling
- Admin metrics aggregation and read APIs
- Privacy-safe analytics collection
- Closed-beta allowlist or override hooks if needed

### Firebase Responsibilities
Firebase is responsible for:

- Authentication
- User profile documents
- User settings documents
- Cloud cycle metadata
- Cloud cycle payload storage
- Subscription entitlement mirrors used by the client

### Local-First Data Model
- Local auto-save remains the fallback for everyone
- Premium cloud sync layers on top of the current forecast serialization model instead of replacing it
- Existing local JSON save/export formats remain supported
- Premium expiration should not delete a user's local work
- Premium expiration defaults to **read-only cloud access** in v1.4.0: no new cloud saves or background sync, but no automatic deletion workflow is introduced in this release

## Phase Order

### Phase 1: Foundation and Trust
Establish release framing, legal/privacy updates, hosting contracts, feature flags, and privacy-safe analytics behavior. Update the roadmap so the repo clearly reflects the new v1.4.0 direction.

### Phase 2: Accounts and Free Settings Sync
Add optional accounts, account entry points, and synced profile/settings without breaking the current signed-out local workflow.

### Phase 3: Stripe Billing and Entitlements
Add pricing, checkout, billing portal, webhooks, and client entitlement state so premium features can be gated cleanly.

### Phase 4: Premium Cloud Cycle Library
Add hosted cycle storage, cloud library management, load/save flows, and debounced sync for premium-backed work.

### Phase 5: User Metrics and Admin Dashboard
Ship progress-only account metrics plus a minimal private admin dashboard using aggregate anonymous telemetry.

### Phase 6: Beta Hardening and Launch
Run the closed beta, tighten docs and messaging, verify regressions, and launch with the two-week annual promo window.

## Phase Dependencies
- Phase 1 must land first because it defines the trust, policy, and hosting contract for every later phase.
- Phase 2 depends on Phase 1 environment and privacy decisions.
- Phase 3 depends on Phase 2 identity flows.
- Phase 4 depends on Phases 2 and 3 because cloud save/load requires both identity and entitlements.
- Phase 5 depends on Phases 1-4 event and data model decisions.
- Phase 6 depends on all previous phases being merged and validated on `beta`.

## Release Gates
Before v1.4.0 can be promoted beyond beta:

- Updated roadmap, release planning docs, ToS, and privacy policy are in place
- Signed-out free workflows still work end-to-end
- Account creation and provider sign-in work reliably
- Stripe entitlements update correctly from webhook events
- Premium cloud save/load works without corrupting the current forecast schema
- Admin metrics are aggregate-only and do not retain raw identifying user data as product analytics
- Closed beta completes with issue triage and at least one polish pass
- Build and test checks pass on the release candidate branch

## Rollout Plan
1. Ship foundation work and account scaffolding behind hosted feature flags.
2. Open a closed beta for highly active users and trusted friends.
3. Observe sign-in, billing, sync, and support pain points before public launch.
4. Launch premium publicly with `$3/month`, `$25/year` intro pricing, and `$30/year` standard annual pricing configured to activate after the two-week window.
5. Keep messaging centered on "hosted sync pays for infrastructure" rather than "core features are now paid."

## Explicit Deferrals
These ideas are intentionally deferred past v1.4.0:

- User-facing verification score/ranking UI
- Leaderboards or public profile comparison
- Real-time collaboration
- Shared/team workspaces
- Deep analytics segmentation beyond high-level product health
- Automated data deletion lifecycle for expired premium storage
- WarnGen implementation work
