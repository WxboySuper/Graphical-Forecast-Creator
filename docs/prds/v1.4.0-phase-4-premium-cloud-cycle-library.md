# v1.4.0 Phase 4 PRD: Premium Cloud Cycle Library

**Execution target:** one implementation branch and one PR

## Objective
Ship the premium cloud save/load experience for forecast cycles and discussions while preserving the current local-first workflows and save schema.

## Why Now
This is the core monetized feature of v1.4.0. Once identity and entitlements exist, GFC needs a polished hosted storage workflow that feels useful enough to support the subscription without compromising the free local product.

## In Scope
- Cloud-backed cycle save/load/delete/rename
- Cloud library UI
- Forecast-cycle payload storage using the existing serialized data model
- Discussion persistence as part of the cycle payload
- Debounced background sync for cloud-backed cycles
- Local fallback and recovery behavior
- Read-only handling when premium is inactive

## Out of Scope
- Real-time collaboration
- Shared folders or team workspaces
- Branching/merge conflict resolution between multiple cloud edits
- Automatic migration of every local saved cycle into the cloud

## Implementation Checklist
- [ ] Reuse the current forecast serialization format for cloud payloads instead of creating a second document shape
- [ ] Store cloud cycle metadata separately from the full payload so the library can list items quickly
- [ ] Add a `/cloud` or equivalent cloud-library surface for premium users
- [ ] Let users create a new cloud-backed save from the current working cycle
- [ ] Let users load, rename, and delete cloud-backed cycles
- [ ] Persist discussions inside the same cloud cycle payload for a single restore path
- [ ] Add debounced background sync only for the currently opened cloud-backed cycle
- [ ] Keep local auto-save active for everyone, including premium users
- [ ] Show sync state clearly: saving, saved, error, offline, or newer cloud version detected
- [ ] Treat expired premium as read-only cloud access in v1.4.0: loading remains available, new cloud writes and background sync are disabled

## Test Checklist
- [ ] Premium user can create, load, rename, and delete a cloud cycle
- [ ] Discussion text restores correctly with the loaded cycle
- [ ] Large but normal real-world forecast payloads save and load without schema drift
- [ ] Offline or failed sync does not destroy the current local working state
- [ ] A signed-out or non-premium user cannot write to the cloud library
- [ ] Premium-expired account sees read-only behavior instead of destructive loss
- [ ] Existing local save/export flows still work after cloud features are added

## Rollout Notes
- Closed beta users should test this with real forecasting workflows across more than one device
- The library UI should be framed as hosted convenience, not as the new default save path for everyone
- If sync stability is shaky, prefer manual cloud save plus explicit status over pretending the experience is realtime-perfect

## Dependencies / Blockers
- Depends on Phase 2 accounts and Phase 3 entitlements
- Needs Firebase storage/data-model decisions from the release plan implemented
- Metrics work in Phase 5 depends on the final save/load event shapes from this phase
