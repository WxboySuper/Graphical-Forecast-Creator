# v1.4.0 Phase 1 PRD: Foundation and Trust

**Execution target:** one implementation branch and one PR

## Objective
Prepare GFC for hosted accounts and premium sync by locking the release framing, legal/privacy docs, roadmap, configuration model, and privacy-safe telemetry foundation.

## Why Now
Every later phase depends on users trusting the product direction. Before shipping auth, billing, or cloud storage, the repo and hosted service need to clearly explain what is free, what is paid, what data is collected, and how hosted features are enabled or disabled.

## In Scope
- Update `ROADMAP.md` so v1.4.0 is no longer advertised as WarnGen
- Add or update release-facing documentation for v1.4.0 scope and promises
- Add a privacy policy suitable for hosted accounts, billing, and aggregate analytics
- Update ToS/billing language where needed for hosted service behavior
- Define hosted feature flags and capability checks
- Convert current analytics expectations to privacy-safe aggregate collection rules
- Define server/Firebase responsibility boundaries in docs and config scaffolding

## Out of Scope
- User sign-in UI
- Firebase Auth implementation
- Stripe checkout flows
- Cloud save/load UI
- Account metrics screens

## Implementation Checklist
- [ ] Rewrite the post-v1.0 roadmap sections to reflect shipped `v1.1.0`-`v1.3.0` work and the new `v1.4.0` milestone
- [ ] Add a privacy policy document that explains accounts, billing, storage, and anonymous metrics
- [ ] Update ToS or related hosted-service copy for accounts, subscriptions, and storage expectations
- [ ] Introduce env-driven hosted feature flags for local-only, auth-only, and full hosted modes
- [ ] Document the self-host fallback so open-source operators can provide their own Firebase/Stripe config or disable hosted features entirely
- [ ] Replace raw-IP analytics goals with aggregate anonymous event requirements and short-lived operational logging only
- [ ] Expand `server/` documentation to cover billing, admin metrics, and analytics responsibilities
- [ ] Add implementation notes for how premium expiration should behave in v1.4.0: disable new cloud writes, keep existing local work safe, and avoid automatic cloud deletion workflows

## Test Checklist
- [ ] Build succeeds with hosted features disabled
- [ ] Build succeeds with Firebase-only hosted flags enabled
- [ ] Build succeeds with Firebase + Stripe hosted flags enabled
- [ ] Documentation links resolve correctly from roadmap, release docs, and PRDs
- [ ] No existing signed-out app flow is blocked by missing hosted configuration

## Rollout Notes
- This phase can merge before any hosted feature is visible to end users
- Privacy and billing copy must land before closed beta invitations go out
- If analytics behavior changes, ensure the old raw-IP collection path is not treated as the long-term production model

## Dependencies / Blockers
- Final pricing and provider choices are already locked in the release plan
- Needs access to the current hosted deployment assumptions so the env/config docs match reality
- If legal wording becomes a blocker, ship the technical config scaffolding first but do not open beta until the docs are complete
