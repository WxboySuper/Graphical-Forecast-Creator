# v1.4.0 Phase 6 PRD: Beta Hardening and Launch

**Execution target:** one implementation branch and one PR

## Objective
Run the closed beta, harden the hosted account and premium workflows, complete launch messaging, and release v1.4.0 with a stable free/premium boundary.

## Why Now
This release introduces new trust-sensitive behaviors: sign-in, billing, storage, and analytics. It needs a deliberate beta and launch pass, not just a final merge.

## In Scope
- Closed-beta rollout
- Regression fixes from beta feedback
- Launch copy and release notes
- Pricing/promo readiness
- Final validation on `beta` before promotion

## Out of Scope
- New major features outside the existing six-phase plan
- WarnGen implementation
- Post-launch retention experiments

## Implementation Checklist
- [ ] Create the closed-beta tester list and enable hosted features only for that group as needed
- [ ] Gather focused feedback on sign-in, pricing clarity, cloud save/load reliability, and perceived fairness
- [ ] Fix release-blocking issues found during beta before broad launch
- [ ] Write release notes, support-post copy, and pricing explanation copy that reinforce the free-vs-premium boundary
- [ ] Confirm the annual promo switches from `$25/year` to `$30/year` through configuration, not a manual code edit
- [ ] Verify production env values for Firebase, Stripe, admin access, and hosted feature flags
- [ ] Run the final build/test pass on the release candidate
- [ ] Promote through the repo's existing `beta` to `main` flow only after checks and manual confidence are in place
- [ ] Move completed v1.4.0 PRDs into `docs/prds/done/` as the release work finishes

## Test Checklist
- [ ] Full hosted sign-in flow works on beta
- [ ] Free local-only workflow still works on beta when hosted features are unavailable or disabled
- [ ] Premium checkout, entitlement update, and cloud save/load work on beta
- [ ] User and admin metrics look sane with beta data
- [ ] `pnpm run build` passes on the release candidate
- [ ] Relevant automated tests pass on the release candidate
- [ ] Manual smoke test covers home, forecast, discussion, verification, account, pricing, cloud library, and admin routes

## Rollout Notes
- Beta feedback should be triaged into blockers, launch-soon polish, and post-v1.4.0 follow-ups
- Launch messaging must stay grounded: premium supports hosted sync/storage, not access to the core craft of forecasting
- If a launch-critical hosted flow is unstable, delay launch rather than weakening the user's trust

## Dependencies / Blockers
- Depends on Phases 1-5 being merged into `beta`
- Needs live hosted configuration and access to the beta deployment
- Promotion should wait for both technical stability and message clarity
