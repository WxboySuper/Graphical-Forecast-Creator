# v1.4.0 Phase 5 PRD: User Metrics and Admin Dashboard

**Execution target:** one implementation branch and one PR

## Objective
Add progress-first account metrics for users and a minimal private admin dashboard that tells the story of GFC's growth without collecting invasive personal data.

## Why Now
Once accounts, billing, and hosted storage exist, both users and the maintainer need basic visibility. Users should feel progress and continuity; the maintainer should be able to understand adoption, premium uptake, and hosted-service health without drifting into surveillance.

## In Scope
- Account-facing progress metrics
- Anonymous aggregate admin metrics
- Minimal private admin route
- Event taxonomy needed to calculate those metrics

## Out of Scope
- Public leaderboards
- User-facing verification score/rank UI
- Deep marketing attribution
- Per-user admin drill-down beyond access-control basics
- Raw IP-based product analytics

## Implementation Checklist
- [ ] Define a small event taxonomy for account actions, cloud actions, and hosted-product activity
- [ ] Count an "active day" when a signed-in user completes a meaningful action such as saving a cycle, editing a discussion, or running verification
- [ ] Add progress-only user metrics: active-day streak, total active days, cycles created, cloud cycles saved, discussions written, and verification sessions run
- [ ] Add an account metrics panel under `/account`
- [ ] Introduce an anonymous installation or session identifier strategy that supports aggregate active-device metrics without storing user-generated forecast content in analytics events
- [ ] Aggregate admin metrics by day for active devices, signed-in accounts, premium subscriptions, signups, upgrades, cloud saves/loads, storage usage, and cancellations
- [ ] Add a private `/admin` dashboard with headline counts and simple 7-day / 30-day trends
- [ ] Restrict admin access with an explicit allowlist and server-side checks
- [ ] Ensure analytics retention and storage match the privacy-first promise from Phase 1

## Test Checklist
- [ ] User metrics increment only on qualifying actions
- [ ] Streak logic handles same-day repeated activity without double-counting
- [ ] Admin route is blocked for non-admin users
- [ ] Admin metrics show aggregate values without exposing stored forecast payload contents
- [ ] Active-device logic works without relying on long-term raw IP storage
- [ ] Hosted analytics can be disabled or degraded safely if config is incomplete

## Rollout Notes
- Keep the first admin dashboard intentionally small; Firebase and Stripe consoles remain the deeper truth
- If any metric feels privacy-invasive during beta, remove it instead of weakening the privacy promise
- User-facing metrics should feel motivational, not judgmental

## Dependencies / Blockers
- Depends on Phases 2-4 for identity, entitlement, and cloud action events
- Requires final agreement on admin allowlist configuration
- Release should not launch with metrics that contradict the privacy guarantees documented in Phase 1
